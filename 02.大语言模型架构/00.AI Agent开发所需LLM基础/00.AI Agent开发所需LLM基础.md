# 00.AI Agent 开发所需 LLM 基础

这一节只保留 AI Agent 应用开发最需要掌握的 LLM 基础。目标不是做模型算法工程师，而是理解大模型在工程系统里如何输入、生成、检索、受控和评估。

## 1. LLM 的本质：基于上下文预测 token

LLM 可以先理解为一个 **next-token predictor**：

```text
已有上下文 tokens
↓
预测下一个 token 的概率分布
↓
根据解码策略选择 / 采样一个 token
↓
把新 token 加回上下文
↓
继续预测下一个 token
```

所以模型不是一次性“匹配完整答案”，而是逐 token 生成。

例如：

```text
输入上下文：RAG 是一种

候选下一个 token：
检索：0.60
结合：0.25
方法：0.10
系统：0.05
```

模型输出哪个 token，取决于概率分布和解码策略。

对 Agent 开发来说，这意味着：

```text
Prompt 决定上下文
上下文影响模型下一步预测
工具调用、JSON 输出、计划步骤，本质上都是让模型预测出符合约束的 token 序列
```

## 2. Token、上下文窗口、成本和延迟

LLM 不直接处理文本，而是先通过 tokenizer 把文本变成 token id：

```text
用户文本
↓
Tokenizer
↓
token ids
↓
Embedding
↓
Transformer
```

### 2.1 Token 是什么？

Token 是模型处理文本的基本单位。它可能是一个字、一个词、一个子词、一个标点，甚至是字节片段。

```text
中文：RAG 是什么？
可能切成：RAG / 是 / 什么 / ？

英文：retrieval-augmented generation
可能切成：retrieval / - / augmented / generation
```

不同模型的 tokenizer 不同，因此同一段文本在不同模型中 token 数可能不同。

### 2.2 上下文窗口是什么？

上下文窗口表示模型一次请求最多能处理多少 token，包括：

```text
system prompt
user message
history messages
RAG 检索内容
tool call / tool result
assistant output
```

如果超过上下文窗口，就需要截断、摘要、压缩历史或减少检索内容。

### 2.3 成本和延迟怎么来？

工程上主要关注：

```text
输入 token 数：影响 prompt 成本和首 token 延迟
输出 token 数：影响生成成本和总耗时
上下文长度：越长 attention / KV cache 压力越大
模型大小：越大通常越贵、越慢，但能力更强
工具调用次数：Agent 多轮循环会放大延迟
```

Agent 系统里，成本和延迟经常不是单次 LLM 调用决定的，而是由多轮流程叠加出来：

```text
规划 -> 检索 -> 调工具 -> 观察结果 -> 再规划 -> 生成答案
```

## 3. Temperature、top-p、top-k 对稳定性的影响

LLM 每一步都会产生一个下一个 token 的概率分布。解码参数决定“怎么从概率分布里选 token”。

### 3.1 Temperature

`temperature` 控制随机性：

```text
低 temperature：更稳定、更保守、更可复现
高 temperature：更多样、更发散、更有创意
```

Agent 场景通常需要低随机性：

```text
工具调用：0 ~ 0.3
JSON / 结构化输出：0 ~ 0.3
事实问答 / RAG：0.2 ~ 0.7
头脑风暴 / 文案：0.8+
```

### 3.2 top-k

`top-k` 表示每一步只在概率最高的 k 个候选 token 中采样。

```text
k = 1：近似贪心，只选最高概率 token
k 越大：候选范围越大，输出越多样
```

### 3.3 top-p

`top-p` 也叫 nucleus sampling，表示只在累计概率达到 p 的候选 token 中采样。

```text
top-p = 0.9：
从概率最高的 token 开始累加，直到累计概率达到 0.9，只在这部分 token 中采样
```

### 3.4 Agent 开发中的经验

```text
需要稳定、可解析、可执行：降低 temperature / top-p
需要创意、多样表达：提高 temperature / top-p
工具调用失败多：优先降低随机性，并加强 schema 约束
```

## 4. Embedding：把文本变成向量

Embedding 在 LLM 内部和应用层都有使用。AI Agent 开发更常关注**应用层 embedding**：

```text
文本
↓
Embedding Model
↓
向量
```

向量可以用于计算语义相似度：

```text
用户问题向量
文档片段向量
↓
相似度计算
↓
召回相关文档
```

常见用途：

```text
搜索：语义检索、RAG 召回
聚类：把相似文本分组
推荐：找相似内容、相似商品、相似问题
分类：用向量特征做文本分类
去重：判断两段文本是否语义重复
```

在 RAG 中，embedding 的位置通常是：

```text
文档切分
↓
为每个 chunk 生成 embedding
↓
写入向量数据库
↓
用户问题生成 embedding
↓
检索相似 chunk
↓
把检索结果放进 prompt
↓
LLM 生成答案
```

需要注意：

```text
Embedding 负责“找相关内容”
LLM 负责“基于上下文生成答案”
```

## 5. 幻觉的来源

幻觉可以理解为：

> LLM 在缺少可靠上下文或外部校验时，生成了统计上合理但事实上错误的内容。

常见来源：

### 5.1 训练知识过期

模型训练完成后，参数里的知识不会自动更新。

```text
最新政策
最新 API
最新价格
公司内部文档
私有系统状态
```

这些内容不能只依赖模型记忆，需要检索、工具或数据库。

### 5.2 上下文不足

用户问题太短、缺少约束、缺少背景时，模型会基于常见模式补全。

```text
“这个报错怎么解决？”
“帮我优化一下”
“这个接口怎么用？”
```

如果没有代码、日志、文档，模型很容易猜。

### 5.3 检索证据错误

RAG 不是天然可靠。检索可能召回：

```text
不相关文档
过期文档
相似但不适用的文档
片段缺少关键上下文
```

如果错误证据进入 prompt，模型可能会基于错误证据生成看似可靠的答案。

### 5.4 模型过度补全

LLM 的训练目标是预测下一个 token，不是天然保证“只说确定的事实”。

当它不知道答案时，也可能生成一个语言上顺畅的回答。

## 6. Agent 开发中的缓解方式

```text
RAG：把真实资料放进上下文
Tool Calling：查数据库、调用 API、跑代码、读文件
引用来源：要求答案绑定证据
结构化输出：用 JSON Schema / function schema 约束输出
低随机性：降低 temperature，减少发散
拒答策略：证据不足时明确说不知道
校验器：对输出做事实、格式、权限、业务规则检查
日志与评估：记录 prompt、检索结果、工具调用和失败样本
```

一句话总结：

> AI Agent 不是只靠 LLM “聪明”就能稳定工作，而是通过上下文、工具、检索、约束和校验，把 next-token predictor 组织成可靠的软件系统。
